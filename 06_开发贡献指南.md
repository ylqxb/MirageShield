# 开发贡献指南

## 贡献流程

### 1. 环境准备

**克隆代码库**：
```bash
git clone https://github.com/ylqxb/MirageShield.git
cd MirageShield
```

**创建开发环境**：
```bash
# 创建虚拟环境
python3 -m venv venv

# 激活虚拟环境
# Windows
venv\Scripts\activate
# Linux
source venv/bin/activate

# 安装依赖
pip install -r requirements.txt

# 安装开发依赖
pip install pytest flake8 black
```

### 2. 代码贡献

**创建分支**：
```bash
# 从 develop 分支创建新分支
git checkout develop
git pull
git checkout -b feature/your-feature-name
```

**提交代码**：
```bash
# 检查代码风格
flake8 .

# 格式化代码
black .

# 运行测试
python -m pytest

# 提交代码
git add .
git commit -m "feat: 描述你的功能"
git push origin feature/your-feature-name
```

**创建 Pull Request**：
1. 访问 GitHub 仓库
2. 点击 "Pull requests" 标签
3. 点击 "New pull request" 按钮
4. 选择你的分支和目标分支
5. 填写 Pull Request 描述
6. 点击 "Create pull request" 按钮

### 3. 代码审查

**审查流程**：
1. 提交 Pull Request 后，项目维护者会进行代码审查
2. 审查者会提出修改意见
3. 你需要根据意见修改代码
4. 修改后，再次提交代码
5. 审查通过后，代码会被合并

**审查标准**：
- 代码正确性和完整性
- 代码风格和规范
- 性能和安全性
- 文档和注释

## 开发规范

### 1. 代码风格

**Python 代码风格**：
- 遵循 PEP 8 代码风格
- 使用 4 空格缩进
- 每行代码不超过 80 字符
- 变量名使用小写字母和下划线
- 函数名使用小写字母和下划线
- 类名使用驼峰命名法

**JavaScript 代码风格**：
- 遵循 JavaScript 标准风格
- 使用 2 空格缩进
- 每行代码不超过 80 字符
- 变量名和函数名使用 camelCase
- 类名使用 PascalCase

**CSS 代码风格**：
- 使用 kebab-case 命名类名
- 使用 2 空格缩进
- 每行代码不超过 80 字符
- 遵循 BEM 命名规范

### 2. 命名规范

**文件命名**：
- Python 文件：使用小写字母和下划线，如 `network_scan.py`
- JavaScript 文件：使用小写字母和下划线，如 `system_status.js`
- CSS 文件：使用小写字母和下划线，如 `main_style.css`

**变量命名**：
- 局部变量：使用小写字母和下划线，如 `network_scan_results`
- 全局变量：使用大写字母和下划线，如 `MAX_RETRY_COUNT`
- 常量：使用大写字母和下划线，如 `ENCRYPTION_KEY`

**函数命名**：
- 普通函数：使用小写字母和下划线，如 `scan_network()`
- 类方法：使用小写字母和下划线，如 `get_status()`
- 私有方法：使用双下划线前缀，如 `__private_method()`

**类命名**：
- 使用驼峰命名法，如 `NetworkScanner`
- 类名应描述其功能

### 3. 文档规范

**文档字符串**：
- 每个函数和类都应有文档字符串
- 文档字符串应包含功能描述、参数说明和返回值
- 遵循 Google 风格的文档字符串

**代码注释**：
- 复杂的代码应添加注释
- 注释应清晰描述代码的功能
- 避免冗余的注释

**README 文件**：
- 每个模块应有 README 文件
- README 文件应描述模块的功能和使用方法

### 4. 测试规范

**单元测试**：
- 每个函数和类都应有单元测试
- 测试应覆盖核心功能和边界情况
- 使用 pytest 框架编写测试

**集成测试**：
- 测试不同模块之间的交互
- 验证系统的整体功能

**测试覆盖率**：
- 代码覆盖率应达到 80% 以上
- 重要功能的覆盖率应达到 100%

## 分支管理

### 1. 分支策略

**主分支**：
- `master`：稳定版本，用于生产环境
- `develop`：开发分支，用于集成新功能

**功能分支**：
- `feature/*`：新功能开发
- `fix/*`：Bug 修复
- `docs/*`：文档更新
- `chore/*`：构建或依赖更新

### 2. 分支流程

**功能开发**：
1. 从 `develop` 分支创建功能分支
2. 在功能分支上开发
3. 提交代码并创建 Pull Request
4. 代码审查通过后，合并到 `develop` 分支

**Bug 修复**：
1. 从 `master` 分支创建修复分支
2. 在修复分支上修复 Bug
3. 提交代码并创建 Pull Request
4. 代码审查通过后，合并到 `master` 和 `develop` 分支

**版本发布**：
1. 从 `develop` 分支创建发布分支
2. 在发布分支上进行版本更新和测试
3. 测试通过后，合并到 `master` 分支
4. 标签版本号
5. 合并回 `develop` 分支

## 提交规范

### 1. 提交信息格式

**基本格式**：
```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型**：
- `feat`：新功能
- `fix`：Bug 修复
- `docs`：文档更新
- `style`：代码风格修改
- `refactor`：代码重构
- `test`：测试更新
- `chore`：构建或依赖更新

**范围**：
- 模块名称，如 `api`、`agents`、`control_plane` 等

**主题**：
- 简短描述，不超过 50 字符
- 使用祈使句，如 "Add feature" 而不是 "Added feature"

**正文**：
- 详细描述，可多行
- 描述为什么要做这个修改
- 描述如何实现这个修改

**页脚**：
- 关联的 Issue 编号，如 `Closes #123`
- 破坏性变更，如 `BREAKING CHANGE: ...`

### 2. 提交示例

**新功能**：
```
feat(api): 添加系统状态 API 接口

添加了 /api/system/status 接口，用于获取系统运行状态
包括智能体状态、防护级别和系统资源使用情况

Closes #123
```

**Bug 修复**：
```
fix(agents): 修复智能体连接失败问题

修复了智能体在网络不稳定时连接失败的问题
通过增加重连机制和超时处理

Closes #456
```

**文档更新**：
```
docs: 更新 API 文档

更新了 API 文档，添加了新接口的说明
修正了一些错误的描述

Closes #789
```
